Intent-based access control for Kafka in Kubernetes

We can realize intent-based access control in Kubernetes by leveraging the built-in capabilities Kafka offers for enforcement using its ACL mechanism, coupled with K8s identities Otterize can easily bake intro mTLS certificates.

Of course, Otterize eliminates all of that. Let's see how.

Choose a Kubernetes cluster, Kafka and services​

You can use this guide with your own set of services, e.g. first in a Kubernetes cluster running dev or lab services, and eventually in a staging or production cluster. You'll need admin access.

You can also use an existing Kafka deployment or use our provided deployment for ease of use.

Install Otterize​

note

For this tutorial, we'll disable the network policy component, to focus on the Kafka ACL authorization mechanism. If you already have Otterize installed, redeploy it as shown below. You can always re-enable the network policy component later by deploying it as shown below but without the --set flag.

Use Helm to install the latest version of Otterize:

helm repo add otterize https://helm.otterize.com
helm repo update
helm install -n otterize-system --create-namespace \
--set intentsOperator.operator.enableNetworkPolicyCreation=false otterize otterize/otterize-kubernetes

You can add the --wait flag for Helm to wait for deployment to complete and all pods to be Ready, or manually watch for all pods to be Ready using kubectl get pods -n otterize-system -w.

Configure Kafka for secure access​

To configure a Kafka broker for secure access, we need to configure the broker itself as well as adding Otterize-specific annotations to indicate that we want mTLS credential generation to happen. For the complete list of annotation parameters please follow the SPIRE integration operator documentation.

The following example shows how to configure Bitnami's popular Kafka chart:

  1. Configure super users - add the broker itself and the Otterize Intents Operator:

    superUsers: "User:CN=kafka.kafka,O=SPIRE,C=US;User:CN=intents-operator-controller-manager.otterize,O=SPIRE,C=US"

  2. Configure the Kafka broker's listeners to use SSL

    listeners:
      - "CLIENT://:9092"
      - "INTERNAL://:9093"
    advertisedListeners:
      - "CLIENT://:9092"
      - "INTERNAL://:9093"
    listenerSecurityProtocolMap: "INTERNAL:SSL,CLIENT:SSL"

  3. Set the authentication parameters to

    • Use mTLS
    • Having the credentials stored in a Java KeyStore (JKS) format stored in a K8s secret called kafka-tls-secret with a password set to password.
    • Deny anonymous access
    auth: 
      clientProtocol: mtls
      interBrokerProtocol: mtls
      tls:
        type: jks
        existingSecrets:
          - kafka-tls-secret
        password: password
    authorizerClassName: kafka.security.authorizer.AclAuthorizer
    allowEveryoneIfNoAclFound: true

  4. Add pod annotations to let Otterize know it needs to generate the JKS credentials and store them at the specified K8s secret:

    podAnnotations:
      spire-integration.otterize.com/cert-type: jks
      spire-integration.otterize.com/tls-secret-name: kafka-tls-secret
      spire-integration.otterize.com/truststore-file-name: kafka.truststore.jks
      spire-integration.otterize.com/keystore-file-name: kafka-keystore.jks
      spire-integration.otterize.com/dns-names: "kafka-0.kafka-headless.kafka.svc.cluster.local,kafka.kafka.svc.cluster.local"
      spire-integration.otterize.com/cert-ttl: "31536000" # 1 year
Expand to see the complete YAML of this example
# Configure Otterize as a super user to grant it access to configure ACLs
superUsers: "User:CN=kafka.kafka,O=SPIRE,C=US;User:CN=intents-operator-controller-manager.otterize,O=SPIRE,C=US"
# Use TLS for the Kafka listeners (Kafka calls them SSL)
listeners:
  - "CLIENT://:9092"
  - "INTERNAL://:9093"
advertisedListeners:
  - "CLIENT://:9092"
  - "INTERNAL://:9093"
listenerSecurityProtocolMap: "INTERNAL:SSL,CLIENT:SSL"
# Annotations for Otterize to generate credentials
podAnnotations:
  spire-integration.otterize.com/cert-type: jks
  spire-integration.otterize.com/tls-secret-name: kafka-tls-secret
  spire-integration.otterize.com/truststore-file-name: kafka.truststore.jks
  spire-integration.otterize.com/keystore-file-name: kafka-keystore.jks
  spire-integration.otterize.com/dns-names: "kafka-0.kafka-headless.kafka.svc.cluster.local,kafka.kafka.svc.cluster.local"
# Authenticate clients using mTLS
auth:
  clientProtocol: mtls
  interBrokerProtocol: mtls
  tls:
    type: jks
    existingSecrets:
      - kafka-tls-secret
    password: password
authorizerClassName: kafka.security.authorizer.AclAuthorizer
# Allocate resources
resources:
  requests:
    cpu: 50m
    memory: 256Mi

The following commands will deploy a Kafka cluster configured to use Otterize-provided mTLS credentials with permissions for Otterize to configure ACLs:

helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm install -n kafka \
--version 14.x.x \
-f https://docs.otterize.com/code-examples/kafka-mtls/helm/values.yaml --create-namespace kafka bitnami/kafka

You can watch for all pods to be Ready using kubectl get pods -n kafka -w.

Testing secure connection to Kafka​

You can set up an interactive session that securely connects to Kafka by deploying a pod configured to get credentials, and then send that pod scripts to produce and consume messages, using the standard Kafka kafka-console-producer and kafka-console-consumer scripts:

Expand to see the details of the pod YAML configurations
  • namespace.yaml
  • interactive.yaml
  • interactive-configmap.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: otterize-tutorial-kafka-mtls
apiVersion: apps/v1
kind: Deployment
metadata:
  name: interactive
  namespace: ibac-for-kafka
spec:
  selector:
    matchLabels:
      app: interactive
  template:
    metadata:
      labels:
        app: interactive
      annotations:
        spire-integration.otterize.com/tls-secret-name: interactive-credentials-secret
        spire-integration.otterize.com/cert-type: jks
        spire-integration.otterize.com/truststore-file-name: kafka.truststore.jks
        spire-integration.otterize.com/keystore-file-name: kafka.keystore.jks
    spec:
      containers:
        - name: interactive
          image: bitnami/kafka
          command: [ "/bin/sh", "-c", "--" ]
          args: [ "sleep infinity" ]
          volumeMounts:
            - mountPath: /app
              name: interactive-properties
            - name: otterize-credentials
              mountPath: /var/otterize/credentials
              readOnly: true
      volumes:
        - name: interactive-properties
          configMap:
            name: interactive-properties
        - name: otterize-credentials
          secret:
            secretName: interactive-credentials-secret
apiVersion: v1
kind: ConfigMap
metadata:
  name: interactive-properties
  namespace: ibac-for-kafka
data:
  client.properties: |
    security.protocol=SSL
    ssl.keystore.location=/var/otterize/credentials/kafka.keystore.jks
    ssl.keystore.password=password
    ssl.truststore.location=/var/otterize/credentials/kafka.truststore.jks
    ssl.truststore.password=password

  1. Deploy a pod configured for the interactive session connecting to Kafka

    kubectl apply -f https://docs.otterize.com/code-examples/ibac-for-kafka/interactive/all.yaml

  2. Verify that the pod was deployed

    kubectl get pods -n ibac-for-kafka -l app=interactive

    You should see the following output:

    NAME                           READY   STATUS    RESTARTS   AGE
    interactive-5568ff58ff-84ksc   1/1     Running   0          8s

  3. Let's open two interactive terminal windows - one for the producer and one for the consumer, so we can see the messages sent and received in real time:

    Producer terminal
    kubectl exec -n ibac-for-kafka -it deploy/interactive -- bash \
    /opt/bitnami/kafka/bin/kafka-console-producer.sh \
    --bootstrap-server kafka.kafka:9092 \
    -topic my-topic \
    --producer.config /app/client.properties
    Consumer terminal
    kubectl exec -n ibac-for-kafka -it deploy/interactive -- bash \
    /opt/bitnami/kafka/bin/kafka-console-consumer.sh \
    --bootstrap-server kafka.kafka:9092 \
    -topic my-topic \
    --consumer.config /app/client.properties

  4. Write any text you like on the producer terminal, send it with an enter and you will see the messages appear on the consumer terminal:

    Producer terminal
    >Hello there consumer, this is the producer!
    >
    Consumer terminal
    Hello there consumer, this is the producer!

Configure Otterize to manage Kafka ACLs​

To allow Otterize to configure ACLs for Kafka we need to create and apply a KafkaServerConfig. The basic configuration allows Otterize to configure Kafka ACLs upon new applied intents and sets a default allow policy for all topics. We will later show how to add fine-grained control over topics.

apiVersion: k8s.otterize.com/v1alpha1
kind: KafkaServerConfig
metadata:
  name: kafkaserverconfig
  namespace: kafka
spec:
  service:
    name: kafka
  addr: kafka.kafka:9092
  tls:
    certFile: /etc/otterize-spire/svid.pem
    keyFile: /etc/otterize-spire/key.pem
    rootCAFile: /etc/otterize-spire/bundle.pem
  topics:
    - topic: "*"
      pattern: prefix
      clientIdentityRequired: false
      intentsRequired: false

You can apply it with

kubectl apply -f https://docs.otterize.com/code-examples/ibac-for-kafka/kafkaserverconfig.yaml

Setting security scope​

Security scopes are set by setting topic selectors and setting permission levels.

An example configuration would be

topic: "my-topic"
pattern: literal
clientIdentityRequired: true
intentsRequired: true

Topic selectors can be set by using a prefix match or a literal match pattern. Read more here.

Permission levels are set by

  • Unauthenticated - client identity not required
    clientIdentityRequired: false
    intentsRequired: false
  • Authenticated - client identity required
    clientIdentityRequired: true
    intentsRequired: false
  • Authorized - client identity and an intent are required
    clientIdentityRequired: true
    intentsRequired: true

To set a security scope for topic access, define:

  • Default access permission levels for topics
  • Per-topic access permission level

Setting the default access permission levels for topics​

To set the default security scope for topics, use the following topic and pattern:

    - topic: "*"
      pattern: prefix

Setting per-topic access permission level​

To set the security scope for a specific topic, use the following topic and pattern:

    - topic: "TOPIC NAME GOES HERE"
      pattern: literal

Example config​

The following KafkaServerConfig

  • Sets a default allow unauthenticated access to all topics;
  • Restricts access to my-topic to authenticated clients;
  • Restricts access to my-topic-intended for authenticated clients with applied intents:
apiVersion: k8s.otterize.com/v1alpha1
kind: KafkaServerConfig
metadata:
  name: kafkaserverconfig
  namespace: kafka
spec:
  service:
    name: kafka
  addr: kafka.kafka:9092
  tls:
    certFile: /etc/otterize-spire/svid.pem
    keyFile: /etc/otterize-spire/key.pem
    rootCAFile: /etc/otterize-spire/bundle.pem
  topics:
    - topic: "*"
      pattern: prefix
      clientIdentityRequired: false
      intentsRequired: false
    - topic: "my-topic"
      pattern: literal
      clientIdentityRequired: true
      intentsRequired: false
    - topic: "my-topic-intended"
      pattern: literal
      clientIdentityRequired: true
      intentsRequired: true

It will be used in the next section as an example.

Testing security scopes​

Let's run through the example config above and check that Kafka access controls are working as configured:

  • A client that's authenticated but without declared intents:
    • can access my-topic successfully;
    • can't access my-topic-intended.
  • Apply intents for that client to access my-topic-intended...
  • Now that client can access my-topic-intended.

You can set up an interactive session that securely connects to Kafka by deploying a pod configured to get credentials and using the popular kafka-console-producer and kafka-console-consumer scripts. There will be one client (this "interactive" pod) and we will use it from two different windows to get it to produce events (from one window) and then consume them (showing them in the second window).

Declare and apply client intents to access my-topic-intended​

Now we'll declare the client's intents to access my-topic-intended:

apiVersion: k8s.otterize.com/v1alpha1
kind: ClientIntents
metadata:
  name: interactive
  namespace:  ibac-for-kafka
spec:
  service:
    name: interactive
  calls:
    - name: kafka
      namespace: kafka
      type: kafka
      topics:
        - name: my-topic-intended
          operation: all

Apply these intents with:

kubectl apply -f https://docs.otterize.com/code-examples/ibac-for-kafka/interactive/interactive-intents.yaml

Finally, let's again open two interactive terminal windows scripted to access my-topic-intended:

What happens when we apply a security scope​

Security scopes translate directly into Kafka ACLs. You can use the parameters topic, pattern, clientIdentityRequired and intentsRequired to configure them. With those, you can choose on which topics you'd like to require an identity (enforce mTLS), on which you'd like to require intents, and on which you don't require any of those. Read more about Kafka configuration here..

Otterize identities, intents and Kafka ACLs​

In the following example, there is one client (myclient) running in the namespace ibac-for-kafka and an intent to access a single topic (my-topic).

Expand to see the example YAML files
  • deployment.yaml
  • client-intents.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myclient
  namespace: ibac-for-kafka
spec:
  selector:
    matchLabels:
      app: myclient
  template:
    metadata:
      labels:
        app: myclient
      annotations:
        spire-integration.otterize.com/tls-secret-name: myclient-credentials-secret
    spec:
      containers:
        - name: myclient
          image: golang
          command: [ "/bin/sh", "-c", "--" ]
          args: [ "sleep infinity" ]

apiVersion: k8s.otterize.com/v1alpha1
kind: ClientIntents
metadata:
  name: myclient
  namespace:  ibac-for-kafka
spec:
  service:
    name: myclient
  calls:
    - name: kafka
      namespace: kafka
      type: kafka
      topics:
        - name: my-topic
          operation: all

Generating credentials​

By annotating the pod with spire-integration.otterize.com/tls-secret-name: myclient-credentials-secret Otterize automatically generates mTLS credentials for the pod.

The certificate's CN is set to be myclient.ibac-for-kafka. This is determined by taking the Otterize service name and appending the namespace. This value will be used in the following step when applying intents.

You can deploy the client with:

kubectl apply -f https://docs.otterize.com/code-examples/ibac-for-kafka/client-deployment.yaml

Run the following to inspect its certificate:

kubectl get secret -n ibac-for-kafka myclient-credentials-secret -o jsonpath='{.data.svid\.pem}' | \
base64 -d | openssl x509 -in svid.pem -text | grep "CN"

You should see the CN configured correctly

  Subject: C = US, O = SPIRE, CN = client.otterize-tutorial-mtls

Applying intents​

When an intent is applied — in our case, for myclient to access my-topic on Kafka — Otterize automatically generates an ACL:

  • the ACL matches the CN value used for this client: myclient.ibac-for-kafka;
  • the ACL allow access to the intended topic with the intended operation.

You can apply the intents with:

kubectl apply -f https://docs.otterize.com/code-examples/ibac-for-kafka/client-intents.yaml

Run the following command to look through the Kafka logs for ACL changes:

kubectl logs -n kafka statefulset/kafka | grep "Processing Acl change" | grep myclient | tail -n 1

You should see that a Kafka ACL was configured to allow access to my-topic for myclient.ibac-for-kafka:

[2022-09-15 07:41:12,632] INFO Processing Acl change notification for
ResourcePattern(resourceType=TOPIC, name=my-topic, patternType=LITERAL), versionedAcls :
Set(User:ANONYMOUS has DENY permission for operations: ALL from hosts: *,
    User:CN=myclient.ibac-for-kafka,O=SPIRE,C=US has ALLOW permission for operations: ALL from hosts: *)
, zkVersion : 5 (kafka.security.authorizer.AclAuthorizer)

Teardown​

To remove the deployed examples run:

kubectl delete namespace ibac-for-kafka
