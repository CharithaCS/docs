Map your cluster

The Network Mapper allows you to map pod-to-pod traffic within your K8s cluster. This tutorial will guide you through installing Otterize, mapping traffic and tracking changes.

Install Network Mapper​

git clone git@github.com:otterize/network-mapper.git
cd network-mapper
helm upgrade --install -n otterize mapper helm/ -f helm/values.yaml --create-namespace
cd src/cli/cmd/
go build -o otterize .
sudo cp ./otterize /usr/local/bin/
danger

Decide if we show mapped traffic by namespace or not. If not, then the explanation about pod identities will confuse users as the output ignores namespace.

How to​

Show mapped traffic​

You can view mapped traffic by calling the CLI export command. It supports multiple output formats such as intents and JSON. The following example shows the CLI output for a cluster running three pods: checkout, kafka, zookeeper.

You should see similar structured results on your cluster.

  • Plain
  • Intents
  • JSON

  1. Run the list command:

    otterize intents list

  2. You should get a result based on your existing traffic looking like this:

    checkoutservice calls:
      - orderservice

    orderservice calls:
      - kafka

  1. Run the export command:

    otterize intents export

  2. You should get a result based on your existing traffic looking like this:

    apiVersion: k8s.k8s.otterize.com/v1
    kind: ClientIntents
    metadata:
      name: checkoutservice
    spec:
      service:
        name: checkoutservice
      calls:
        - name: orderservice
    ---
    apiVersion: k8s.k8s.otterize.com/v1
    kind: ClientIntents
    metadata:
      name: orderservice
    spec:
      service:
        name: orderservice
      calls:
        - name: kafka

  1. Run the export command:

    otterize intents export --format json

  2. You should get a result based on your existing traffic looking like this:

    [
      {
        "kind": "ClientIntents",
        "apiVersion": "k8s.otterize.com/v1alpha1",
        "metadata": {
          "name": "checkoutservice"
        },
        "spec": {
          "service": {
            "name": "checkoutservice"
          },
          "calls": [
            {
              "name": "orderservice"
            }
          ]
        }
      },
      {
        "kind": "ClientIntents",
        "apiVersion": "k8s.otterize.com/v1alpha1",
        "metadata": {
          "name": "orderservice"
        },
        "spec": {
          "service": {
            "name": "orderservice"
          },
          "calls": [
            {
              "name": "kafka"
            }
          ]
        }
      }
    ]
info

See the Network Mapper documentation for more details about the CLI.

Change traffic -> Sniff&Diff​

One of the benefits for using the network mapper is the ability to track changes over time for communication within your cluster.

  1. Let's save the current state of traffic from the cluster into a file we will compare against later
    otterize intents list > intents-original.txt

  1. And now we can add traffic to the cluster and see how the Network Mapper tracks it. You can do that by deploying our example which consists of two pods: client and server, communicating over HTTP. Deploy example:

    kubectl apply -n otterize-tutorial-mapper -f https://docs.otterize.com/code-examples/network-mapper/all.yaml

  2. Check that the client and server pods were deployed

    kubectl get pods -n otterize-tutorial-mapper

    You should see

    NAME                      READY   STATUS    RESTARTS   AGE
    client-756f7677f8-d6qdq   1/1     Running   0          45s
    server-6698c58cbc-ssxvx   1/1     Running   0          45s

  3. Export the updated observed intents.

    otterize intents list

    You will now see the client and server pods communication in addition to the previously observed traffic.

    client calls:
      - server

    checkoutservice calls:
      - orderservice

    orderservice calls:
      - kafka

  4. We can also compare both output to see the difference. We'll start by saving the updated state to a file with

    otterize intents list > intents-updated.txt

  5. And compare the original file with the updated file using

    diff --color=always -y intents-original.txt intents-updated.txt;echo

    You should see a result looking like

                                          > client calls:
                                          >   - server
                                          >
    checkoutservice calls:                  checkoutservice calls:
      - orderservice                          - orderservice

    orderservice calls:                     orderservice calls:
      - kafka                                 - kafka

What's next​

Now that we observed cluster intents we can use them with the Intents Operator.

Teardown​

To remove the deployed resources run

kubectl delete namespace otterize-tutorial-mapper
helm uninstall -n otterize mapper
